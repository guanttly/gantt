# LLM Prompt 信息结构重构总结

## 问题背景

根据日志分析（`cmd/agents/rostering/logs/app.log`），发现排班Agent在调用LLM时存在两个主要问题：

### 问题1：负向需求过滤不当
- **现象**：日志显示"共 28 人"有负向需求，但其中包含"董卉妍 (ID:staff_21)"等非候选人员
- **实际**：候选人员只有68人，不应该显示非候选人员的负向需求
- **影响**：给LLM提供了大量无关信息，干扰决策

### 问题2：LLM对连班规则理解错误
- **规则**：rule_3和rule_11要求"下夜班"人员必须从前一天"夜班"人员中选择
- **现象**：LLM推理时认为"由于当前无任何历史排班记录...无需强制依赖前日夜班记录"
- **实际**：系统中有shift_4(夜班)和shift_5(下半夜)的排班数据，但信息组织混乱导致LLM没有正确关联

### 根本原因
原有的prompt构建方式类似"堆垃圾"，把所有信息不加筛选、不分逻辑地全部丢给LLM，导致：
1. 信息量大但缺乏结构
2. 相关信息没有明确的逻辑关系
3. LLM难以准确理解上下文关联

## 解决方案

### 核心思路
**不是"强化提示"，而是"重新组织信息结构"**，按照清晰的语义分类组织信息，明确告知AI各类信息的逻辑关系。

### 新的信息结构（6个语义化部分）

#### 第一部分：当前班次和排班需求
- 班次基本信息（名称、ID、时间段、时长）
- 按日期列出的排班需求人数
- 基本约束条件（每日最长工作时间、班次间最短休息时间）

**目的**：明确"要排什么"

#### 第二部分：相关规则
- 所有与当前排班任务相关的规则
- 规则类型、说明、具体内容
- ID映射处理（UUID→短ID）

**目的**：明确"要遵守什么规则"

#### 第三部分：当前班次排班情况
- 仅显示目标班次自身的已排班数据
- 按日期展示：人员名单 + [固定]标记

**目的**：明确"当前班次已经安排了谁"

#### 第四部分：关联班次及排班情况
- 显示与目标班次时间重叠或逻辑相关的其他班次
- 重点：**连班规则需要的前置班次数据**（如：排"下夜班"需要看"夜班"数据）
- 按日期展示各班次的排班情况

**目的**：明确"相关联的班次情况"，支撑连班规则判断

#### 第五部分：候选人员排班情况
- **仅显示候选名单内人员**的时段占用情况
- 标注与目标班次时间冲突的人员（❌标记）
- 明确哪些人在哪些日期不可安排

**目的**：明确"候选人员中谁有时间冲突"

#### 第六部分：候选人员负向需求（关键过滤）
- **【核心修复】仅显示候选人员的负向需求**
- 添加过滤逻辑：`if !candidateStaffIDs[staffID] { continue }`
- 显示请假、回避等不可用信息

**目的**：明确"候选人员中谁因个人需求不可用"

### 任务说明
在6个部分之后，提供清晰的执行要求：
1. 基本约束（工作时长、休息时间、人数限制）
2. 规则遵守（引用第二部分，强调连班规则参考第四部分）
3. 冲突避免（固定排班、时段冲突、负向需求）
4. 优化策略（负载均衡）
5. 特殊情况处理

## 预期效果

### 问题1修复效果
- **之前**：显示"共 28 人"，包括非候选人员（如董卉妍）
- **之后**：仅显示候选名单内的68人中有负向需求的人员
- **验证点**：日志中不应再出现非候选人员的负向需求信息

### 问题2修复效果
- **之前**：信息混乱，LLM误判"无历史排班记录"
- **之后**：第四部分明确展示关联班次（如shift_4夜班的排班数据）
- **验证点**：LLM在推理时应能正确识别前置班次数据，严格执行连班规则

### 整体改善
1. **信息结构清晰**：6个语义化部分，逻辑关系明确
2. **信息精准过滤**：只显示候选人员相关的数据
3. **易于理解**：LLM可以按部分理解上下文，不再"堆垃圾"
4. **可维护性**：代码结构清晰，每个部分职责明确

2. **连班规则测试**：针对"下夜班"等需要连班的场景
   - 验证LLM是否正确参考前置班次数据

3. **边界测试**：
   - 无关联班次的场景
   - 无负向需求的场景
