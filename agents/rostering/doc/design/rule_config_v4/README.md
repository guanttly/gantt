# V4 规则配置管理设计文档

> **版本**: v1.0  
> **创建日期**: 2025-01  
> **关联文档**: [schedule_v4/](../schedule_v4/README.md)  
> **核心目标**: 设计 V4 规则的配置、生成、编辑、迁移全流程

## 1. 背景与目标

### 1.1 背景

V4 排班设计（`schedule_v4/`）定义了 **确定性规则引擎** 和 **V4 规则模型**（Category/SubCategory/AssociationRole），
但没有回答一个关键问题：**规则怎么配置进系统？**

现有 V3 规则管理存在以下问题：

| 问题 | 现状 | 影响 |
|------|------|------|
| **前后端枚举不一致** | 前端 `max_shifts` vs 后端 `maxCount` | 数据映射混乱，Bug 难排查 |
| **规则语义靠文本** | `ruleData` 存自然语言描述 | 排班时 LLM 每次都要重新理解 |
| **缺少分类体系** | 无 Category/SubCategory | 无法区分硬约束/软偏好，规则引擎无法分类处理 |
| **缺少关联角色** | Association 没有 Role 字段 | 无法表达"来源依赖"等复杂关系 |
| **纯手工配置** | 全靠管理员手填表单 | 易出错，难以表达复杂规则 |

### 1.2 目标

1. **LLM 辅助生成**: 用户用自然语言描述规则，LLM 解析为 V4 结构化字段
2. **人工审核确认**: LLM 解析结果必须经人工审核后才保存
3. **手动编辑调整**: 管理员可以手动创建/编辑规则的所有 V4 字段
4. **V3 规则迁移**: 现有 V3 规则无缝升级到 V4 格式
5. **枚举统一**: 前后端使用同一套枚举值

### 1.3 不在范围内

- 规则引擎的运行时执行逻辑（见 `schedule_v4/04_rule_engine.md`）
- 工作流状态机（见 `schedule_v4/05_workflow.md`）
- 排班执行过程中的 LLM 调用

## 4. 设计原则

1. **LLM 只在配置时调用**: 规则解析是一次性操作，保存后排班引擎用结构化数据
2. **人永远有最终决定权**: LLM 解析结果必须经人工确认，人可以修改任何字段
3. **过渡性兼容，面向清理设计**: V3 兼容仅是过渡态，所有兼容代码必须集中隔离、标记 `@deprecated`，确保全量迁移后可一键移除
4. **前后端枚举统一**: 消除现有不一致，统一为后端枚举值
5. **渐进式迁移**: V3 规则可批量自动迁移，也可逐条人工调整
6. **V3 清理友好**: 兼容层通过独立文件/函数隔离，不侵入 V4 核心逻辑，清理时只需删除隔离层

## 5. V3 兼容废弃策略

> ⚠️ **V3 兼容是过渡态，不是终态**。所有 V3 兼容代码必须遵循以下规范，以便全量迁移后快速清理。

### 5.1 代码隔离规范

| 层次 | 隔离方式 | 清理操作 |
|------|---------|----------|
| **后端兼容层** | 独立文件 `v3_compat.go`，所有 `normalizeXxx` 函数集中于此 | 删除 `v3_compat.go` + 移除 Handler 中的调用 |
| **后端模型** | V3 专属字段标记 `// @deprecated V3` 注释 | 搜索 `@deprecated V3` 删除 |
| **前端映射表** | 独立文件 `v3-compat.ts`，所有 V3→V4 映射集中于此 | 删除 `v3-compat.ts` + 移除 import |
| **前端显示兼容** | `RULE_TYPE_LABELS` 中旧值区块用 `// @deprecated V3` 标记 | 删除标记区块 |
| **数据库** | V3 数据修复脚本独立为 `migration/v3_cleanup.sql` | 执行清理 SQL |
| **迁移服务** | 独立包 `internal/migration/` | 迁移完成后整包删除 |
| **MCP 迁移工具** | `preview_migration`、`execute_migration` 独立文件 | 删除文件 + 注销工具 |

### 5.2 清理触发条件

当满足以下所有条件时，执行 V3 清理：

```
✅ 所有 scheduling_rules.version = 'v4'（无 V3 规则残留）
✅ 所有 rule_associations.role != ''（全部有角色）
✅ 后端日志中 normalizeXxx 转换次数连续 7 天为 0
✅ 前端代码中不再使用 v3-compat.ts 中的任何函数
```

### 5.3 一键清理脚本

```bash
#!/bin/bash
# scripts/cleanup_v3_compat.sh
# 全量迁移完成后，一键清理所有V3兼容代码

set -e
echo "🧹 开始清理 V3 兼容代码..."

# 后端
rm -f services/management-service/internal/port/http/v3_compat.go
rm -rf services/management-service/internal/migration/
rm -f mcp-servers/rostering/tool/rule/preview_migration.go
rm -f mcp-servers/rostering/tool/rule/execute_migration.go

# 前端
rm -f frontend/web/src/pages/management/scheduling-rule/v3-compat.ts
rm -f frontend/web/src/pages/management/scheduling-rule/components/RuleMigrationDialog.vue

# 搜索残留的 @deprecated V3 标记
echo "⚠️ 以下位置仍有 @deprecated V3 标记，请手动清理："
grep -rn '@deprecated V3' --include='*.go' --include='*.ts' --include='*.vue' . || echo "✅ 无残留"

echo "✅ V3 兼容代码清理完成"
```
