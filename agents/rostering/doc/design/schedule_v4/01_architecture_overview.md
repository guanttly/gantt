```
用户输入自然语言规则
       │
       ▼
┌─────────────────────┐
│ Management Service  │
│ RuleParserService   │
│   · LLM 语义化解析   │
└──────────┬──────────┘
           │ ParsedRule[]
           ▼
┌─────────────────────┐
│ 第1层: 结构化验证    │ ← 代码（字段完整性、枚举值、数值合理性）
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ 第2层: 回译验证      │ ← 代码（结构化规则 → 自然语言 → 用户确认）
└──────────┬──────────┘
           ▼
┌─────────────────────┐
```

## 3. 上下文层次架构（L1/L2/L3）

V4 继承 V3 的三层递归上下文架构，但在每层注入规则引擎：

```
┌─────────────────────────────────────────────────┐
│ L1: CreateV4Context（总工作流上下文）               │
│                                                  │
│ · 全局数据: AllStaff, AllShifts, AllRules         │
│ · ★新增: RuleOrganization（规则组织结果）           │
│ · ★新增: ShiftExecutionOrder（班次执行顺序）       │
│ · 工作草稿: WorkingDraft                          │
│ · 占位信息: OccupiedSlots[]                       │
└──────────────────────┬──────────────────────────┘
                       │ 深拷贝可写数据，过滤任务范围
                       ▼
┌─────────────────────────────────────────────────┐
│ L2: TaskExecutionContextV4（任务执行上下文）        │
│                                                  │
│ · 从L1继承的只读数据                               │
│ · 任务范围: TargetShifts, TargetDates             │
│ · ★新增: MatchedRules（该任务匹配的规则）          │
│ · ★新增: ShiftDependencies（班次依赖关系）         │
│ · OccupiedSlots（快照，深拷贝）                    │
└──────────────────────┬──────────────────────────┘
                       │ 拆分为单班次上下文
                       ▼
┌─────────────────────────────────────────────────┐
│ L3: ShiftTaskContextV4（单班次任务上下文）          │
│                                                  │
│ · 单班次特定数据: TargetShift, FilteredRules       │
│ · ★新增: RuleEngine 输出                          │
│   - EligibleCandidates[]（合格候选人）             │
│   - ExcludedCandidates[]（排除人员+原因）          │
│   - PreferenceScores（偏好评分）                   │
│   - ConstraintDetails（约束详情）                  │
│ · ★新增: LLMBrief（传给 LLM 的结构化摘要）         │
│ · Result: ShiftScheduleDraft                      │
└─────────────────────────────────────────────────┘
```

**依赖规则**：
1. `engine/` 不依赖 `workflow/`（单向依赖）
2. `engine/` 不依赖 `pkg/ai/`（确定性，不用 LLM）
3. `workflow/schedule_v4/` 依赖 `engine/` + `pkg/ai/`
4. 所有组件依赖 `domain/model/` 和 `domain/service/`

## 5. 关键技术决策

### 5.1 为什么不修改 V3 而是新建 V4？

1. **V3 的 3-LLM 预分析模式深度耦合**：`analyzeDayRules()` 并行调用 LLM-1/2/3，改造成代码化等于重写
2. **V3 的 prompt 构建假设规则是文本**：`buildProgressiveDayPromptWithAnalysis()` 整体基于文本规则设计
3. **降低风险**：V4 独立开发，V3 保持不变，灰度切换

### 5.2 为什么规则引擎独立于工作流？

1. **复用性**：规则引擎可被规则解析服务（验证阶段）复用
2. **可测试性**：引擎独立测试，不需要启动完整工作流
3. **演进性**：引擎可独立升级规则类型，不影响工作流

### 5.3 为什么保留 LLM 做排班选人？

1. **综合权衡不可公式化**：选人需要综合考虑公平性、连续性、团队搭配等软因素
2. **偏好规则的灵活性**：某些偏好难以精确量化
3. **但 LLM 的输入已经大幅简化**：只需从合格候选人中选，不再理解规则

### 5.4 为什么用拓扑排序而不是简单优先级？

```
场景：下夜班人员必须来自前一日上半夜班
  - V3: SchedulingPriority 设置上半夜=1, 下夜=2 → 如果设错就完全错误
  - V4: 拓扑排序自动推导 上半夜→下夜 → 即使不设优先级也能正确执行
```

拓扑排序的优势：
- 自动检测循环依赖（V3 无法检测）
- 依赖关系变更时自动调整顺序（V3 需要手动改优先级）
- 同层级班次可并行处理（V3 串行）
