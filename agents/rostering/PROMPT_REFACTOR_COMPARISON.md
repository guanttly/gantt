# LLM Prompt 重构前后对比示例

## 重构前的问题示例（来自真实日志）

### 问题片段1：负向需求显示了非候选人员

```
**❌ 不可用人员信息（负向需求）**：
以下候选人员因请假、回避等负向需求，在指定日期不可用（共 28 人）：
- 董卉妍 (ID:staff_21) 不可用日期：2025-01-13, 2025-01-14, ...
- 张三 (ID:staff_45) 不可用日期：2025-01-13
...
```

**问题**：董卉妍等人根本不在68人的候选名单中，但仍然显示了其负向需求

### 问题片段2：LLM推理错误

```
reasoning: "...由于当前无任何历史排班记录，也无明确的前日夜班人员名单，
因此无需强制依赖前日夜班记录..."
```

**问题**：系统中明明有shift_4(夜班)和shift_5(下半夜)的排班数据，但LLM没有正确识别

## 重构后的结构示例

### 第一部分：当前班次和排班需求

```markdown
# 第一部分：当前班次和排班需求

**目标班次**：下夜班 (ID:shift_6, 00:00-08:00, 时长:8.0小时)

**排班需求明细**：
- 2025-01-13：需要 2 人
- 2025-01-14：需要 2 人

**基本约束条件**：
- 当日每人最多工作：16.0 小时
- 班次间最短休息：8.0 小时
```

### 第二部分：相关规则

```markdown
# 第二部分：相关规则

共有 3 条与本次排班相关的规则：

**规则 1：下夜班人员优先从夜班选择** (ID: rule_3)
- 类型：PREFER
- 说明：下夜班(00:00-08:00)优先从前一天夜班(22:00-次日06:00)人员中选择
- 内容：下夜班应从前日夜班人员中选择

**规则 2：下夜班人员必须从夜班选择** (ID: rule_11)
- 类型：MUST
- 说明：强制要求：下夜班人员必须从前一天夜班人员中选择
- 内容：下夜班必须从前日夜班安排
```

### 第三部分：当前班次排班情况

```markdown
# 第三部分：当前班次排班情况

**目标班次：下夜班 (ID:shift_6, 00:00-08:00)**

**已排班情况**：当前暂无排班记录
```

### 第四部分：关联班次及排班情况

```markdown
# 第四部分：关联班次及排班情况

以下是与目标班次时间重叠或逻辑相关的其他班次（用于判断连班规则和时间冲突）：

**夜班 (ID:shift_4, 22:00-06:00)**：
  - 2025-01-12: 李四, 王五 [固定]
  - 2025-01-13: 赵六, 钱七

**下半夜 (ID:shift_5, 02:00-08:00)**：
  - 2025-01-12: 孙八 [固定]
  - 2025-01-13: 周九
```

**关键**：这里明确显示了前一天的夜班数据，LLM可以清楚看到2025-01-12有李四、王五，2025-01-13有赵六、钱七

### 第五部分：候选人员排班情况

```markdown
# 第五部分：候选人员排班情况

以下是本次候选人员在目标日期的占用状态，用于判断时间冲突：

**目标班次时间：00:00-08:00**

**时段冲突人员（候选名单内）**：
**2025-01-13**：
  - ❌ 张三 已排: 早班(06:00-14:00)
  - ❌ 李四 已排: 中班(08:00-16:00)

**⛔ 重要：以上标记❌的人员在对应日期存在时间冲突，绝对不可安排！**
```

### 第六部分：候选人员负向需求

```markdown
# 第六部分：候选人员负向需求

**候选人员中有负向需求的人员（共 5 人）**：
  - 王五 (ID:staff_3)：不可用日期 2025-01-13, 2025-01-14
  - 赵六 (ID:staff_8)：不可用日期 2025-01-13
  - 钱七 (ID:staff_12)：不可用日期 2025-01-14
  - 孙八 (ID:staff_15)：不可用日期 2025-01-13
  - 周九 (ID:staff_20)：不可用日期 2025-01-14

**⛔ 重要：以上人员在标注的日期因请假、回避等原因不可用，绝对禁止安排！**
```

**注意**：这里只显示了5个候选人员，而不是之前的28人（包含非候选人员）

### 任务说明

```markdown
---

# 任务说明

**📝 根据以上6个部分的信息，为目标班次生成合理的排班结果**

**执行要求（必须严格遵守）**：

1. **基本约束**：
   - 当日每人最多工作 16.0 小时
   - 班次间至少休息 8.0 小时
   - 不要超过每日需求人数上限

2. **规则遵守**：
   - 严格执行第二部分列出的所有规则
   - 特别注意连班规则（如下夜班需要前一天上夜班），参考第四部分的关联班次数据

3. **冲突避免**：
   - [固定]标记的排班绝对不可修改（见第三、四部分）
   - 绝对不可安排第五部分标记❌的时段冲突人员
   - 绝对不可安排第六部分有负向需求的人员在不可用日期

4. **优化策略**：
   - 优先选择未排班或排班较少的人员
   - 实现负载均衡，避免同一人承担过多班次

5. **特殊情况**：
   - 如任务描述明确指出无需求，应返回空schedule {}
   - 如无法满足所有约束，在响应中明确说明原因
```

## 关键改进点对比

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| **信息组织** | 混乱堆砌，无明确结构 | 6个语义化部分，逻辑清晰 |
| **候选人员过滤** | ❌ 显示所有人员（149人）| ✅ 只显示候选人员（68人）|
| **负向需求** | 显示28人（含非候选人员）| 显示5人（仅候选人员）|
| **关联班次** | 混在一起，难以区分 | 独立第四部分，明确关联 |
| **连班规则支撑** | 信息分散，LLM难以识别 | 第四部分明确前置数据 |
| **冲突标注** | 笼统提示 | ❌明确标记，分部分说明 |
| **可维护性** | 低（代码逻辑混杂）| 高（结构清晰，职责分明）|

## LLM推理效果预期

### 重构前可能的错误推理

```
由于当前无任何历史排班记录，也无明确的前日夜班人员名单，
因此无需强制依赖前日夜班记录，可以从所有候选人员中灵活选择...
```

### 重构后的正确推理

```
根据第二部分规则1和规则11，下夜班必须从前一天夜班人员中选择。
查看第四部分关联班次，2025-01-13的前一天(2025-01-12)夜班有：李四、王五
但根据第六部分，王五在2025-01-13不可用（负向需求）
根据第五部分，李四在2025-01-13有时间冲突（已排中班）
因此无法满足连班规则，需要调整前一天的夜班安排...
```

## 代码关键改动

### 1. 过滤非候选人员（修复问题1）

```go
// 重构前：没有过滤
for staffID, datesMap := range unavailableStaffMap.StaffDates {
    // 直接处理所有人员，包括非候选人员
    unavailableDetails = append(unavailableDetails, ...)
}

// 重构后：添加过滤
candidateStaffIDs := make(map[string]bool)
for _, staff := range staffList {
    candidateStaffIDs[staff.ID] = true
}

for staffID, datesMap := range unavailableStaffMap.StaffDates {
    if !candidateStaffIDs[staffID] {
        continue  // ← 过滤掉非候选人员
    }
    unavailableDetails = append(unavailableDetails, ...)
}
```

### 2. 分离目标班次和关联班次（修复问题2）

```go
// 重构前：混在一起显示
for _, shift := range shifts {
    // 目标班次和关联班次混在一起
    if isTargetShift || isOverlapping {
        // 显示排班...
    }
}

// 重构后：分开两个部分
// 第三部分：只显示目标班次
if targetShift != nil {
    shiftDraft := workingDraft.Shifts[targetShift.ID]
    // 只显示目标班次数据
}

// 第四部分：只显示关联班次
for _, shift := range shifts {
    if shift.ID == targetShift.ID {
        continue  // 跳过目标班次本身
    }
    if e.isShiftTimeOverlap(targetShift, shift) {
        // 显示关联班次数据
    }
}
```

**总结**：重构的核心不是"说得更多"，而是"说得更清楚"。通过语义化的信息组织，让LLM能够：
1. 快速定位需要的信息
2. 理解信息之间的逻辑关系
3. 正确执行规则和约束
