按当前代码核对后，本方案不应继续标记为“待实施”，而应拆成以下状态：

1. Phase 1 已完成：角色边界和入口边界已经形成明确口径，相关设计文档也开始互相引用。
2. Phase 2 基本完成：后台新增独立后台登录入口并在后端拒绝非后台账号，后台菜单与平台账号创建入口也已收口；但仍有少量前端校验和后台范围接口残留需要继续清理。
3. Phase 3 已完成：应用角色查询、应用权限同步、员工绑定视角已经落地，`app:schedule_admin` 已覆盖分组、班次、规则和本科室应用权限管理。

5. Phase 5 已完成：分组、班次、规则、员工应用角色管理服务均保持“仅科室节点可操作”的作用域约束，并已在现有 `/api/v1` 业务路由上补应用权限校验。

本次修正重点：

1. 把“现状”改成代码核对后的真实状态，而不是设计初稿时的预期状态。
2. 补上遗漏的实际影响文件，尤其是后台独立登录入口、平台账号角色收口、web 端应用权限同步相关文件。
3. 修正 Phase 4 / Phase 5 的实现判断：当前 web 仍走普通 JWT 登录态，不应默认把配置能力切到仅 `app token` 可访问的路由上。

## 一、目标

将当前混杂在机构管理后台中的科室排班业务彻底收口到排班应用中，并把角色语义重新拉直：

1. 平台管理员、机构管理员只属于后台账号体系，只管理账号、组织、员工、权限下放与统计展示。
2. 科室管理员不再作为后台账号存在，而是员工绑定的应用角色，统一在排班应用中工作。
3. 排班分组、班次、规则、排班执行、本科室应用权限全部回归排班应用。
4. 机构管理后台不再允许员工账号登录，也不再承载科室业务配置。

## 二、已确认的设计决策

### 2.1 角色边界

| 角色 | 账号归属 | 登录入口 | 负责事项 | 不负责事项 |
| --- | --- | --- | --- | --- |
| 平台管理员 | 后台账号 | 机构管理后台 | 平台级账号、机构开通、订阅、系统配置、统计 | 排班业务配置与执行 |
| 机构管理员 | 后台账号 | 机构管理后台 | 行政组织、员工管理、机构内后台账号、权限下放 | 科室排班分组、班次、规则、排班执行 |
| 科室管理员 | 员工账号 + 应用角色 | 排班应用 | 本科室排班分组、班次、规则、排班执行、本科室应用权限 | 行政组织调整、后台账号管理 |
| 普通员工 | 员工账号 | 排班应用 | 查看个人排班、请假、个人偏好 | 任何管理后台能力 |

### 2.2 科室管理员的承接方式

- 科室管理员统一收口为应用角色 `app:schedule_admin`。
- 科室管理员必须从真实员工中指派，不能再单独创建 `dept_admin` 后台账号承载业务。
- 员工登录排班应用后，如果被授予 `app:schedule_admin`，则在当前科室获得业务配置和发布能力。

### 2.3 登录入口原则

- 机构管理后台只允许后台账号登录。
- 排班应用只允许员工账号登录。
- 员工使用工号、手机号或邮箱登录排班应用，不需要也不应该进入后台。

## 三、目标系统形态

### 3.1 机构管理后台保留内容

保留以下后台能力：

1. 平台级管理：机构、订阅、审计、系统配置。
2. 机构级管理：组织架构、员工管理、后台账号管理。
3. 权限下放：决定机构内哪些员工可以被授予哪些应用角色。
4. 后续统计展示：排班结果、人员利用率、请假概况等只读看板。

后台不再长期承载以下页面或操作：

1. 排班分组管理。
2. 班次管理。
3. 规则管理。
4. 科室排班执行。
5. 本科室应用权限日常授权。

### 3.2 排班应用承接内容

排班应用承接以下科室业务能力：

1. 排班分组管理。
2. 班次配置与维护。
3. 规则配置与维护。
4. 排班创建、执行、调整、发布。
5. 本科室应用权限管理。
6. 普通员工的个人排班与请假。

### 3.3 过渡后的角色关系

```text
后台账号体系
  platform_admin / org_admin
    -> 只进入机构管理后台

员工账号体系
  app:employee
  app:scheduler
  app:schedule_admin
  app:leave_approver
    -> 只进入排班应用
```

## 六、推荐实现顺序

为避免前后端互相阻塞，建议按以下顺序推进：

1. 以当前已完成的 Phase 2 为基线，确认后台登录白名单和后台菜单收口不再回退。
2. 先清理 `internal/auth/rbac.go` 中后台角色对科室业务的权限残留。
3. 扩展 `app:schedule_admin` 权限集合，补齐分组、班次、规则、本科室授权管理能力。
4. 在现有 `/api/v1` 普通 JWT 业务路由上补应用权限校验。
5. 恢复 web 端业务路由、导航和按钮入口。
6. 完成联调与冒烟。
7. 最后处理存量 `dept_admin` 兼容迁移与文档清理。

这样做的原因是：当前 web 端仍走普通 JWT，如果先放开前端入口、后补后端应用权限，会把科室配置能力短暂暴露给普通员工，顺序必须反过来。

2. 审核并收窄 [internal/auth/rbac.go](internal/auth/rbac.go) 中 `org_admin`、`dept_admin` 的业务权限残留。
3. 扩展应用权限枚举与 `app:schedule_admin` 权限集。
4. 为分组、班次、规则、本科室授权接口补充应用权限鉴权，优先落在现有 `/api/v1` 普通 JWT 业务路由上。
5. 仅在整体认证切换成立时，再评估是否新增 app token 专用配置路由。

### 7.2 机构后台任务

1. 移除或隐藏分组、班次、规则、本科室应用权限菜单。
2. 平台账号页不再面向科室业务创建 `dept_admin`。
3. 登录页与空状态文案重写为“后台账号专用”。
4. 清理后台前端残留的 `dept_admin` 本地角色判断和无效跳转。

### 7.3 排班应用任务

1. 恢复分组、班次、规则、本科室应用权限的真实路由与导航入口。
2. 复用现有 web 页面，按应用权限显示导航、页面和按钮。
3. 对旧重定向路径做兼容替换，逐步下线 `management-admin` 提示页。
4. 保持普通员工工作台简洁，不混入后台概念。

### 7.4 文档与测试任务

1. 更新角色设计文档和相关 PRD。
2. 更新端到端冒烟清单。
3. 增加“后台拒绝员工登录”和“应用角色承接科室业务”的测试项。

## 八、风险与处理建议

### 风险 1：存量 `dept_admin` 账号无法平滑退场

建议：

1. 先停止创建新账号。
2. 给旧账号设计迁移标识和提示页。
3. 在数据准备完成后再彻底移除。

### 风险 2：web 页面恢复后，后端鉴权尚未打通

建议：

1. 先在 web 当前使用的普通 JWT 业务路由上补应用权限校验。
2. 前端页面上线前先跑联调用例。
3. 不要把“切 app token”与“恢复 web 配置入口”绑成一次上线。

### 风险 3：应用权限与节点角色语义继续混用

建议：

1. 明确节点角色只决定后台身份。
2. 明确应用角色只决定业务能力。
3. 后续新增功能必须先判断属于后台能力还是业务能力。

### 风险 4：`dept_admin` 角色虽然不再作为入口，但仍在系统 RBAC 中长期残留

建议：

1. 先停止新增和登录入口，再逐步清理权限定义与使用方。
2. 在存量账号迁移完成前，不要贸然删除数据库中的系统角色记录。
3. 清理时同步检查后端权限判断、前端角色判断和测试用例，避免只删一层导致隐藏回归。

## 九、完成标准

满足以下条件时，视为本方案落地完成：

1. 机构管理后台不再允许员工账号登录。
2. 科室管理员通过员工账号 + `app:schedule_admin` 在排班应用完成本科室业务管理。
3. 后台只保留行政管理与统计能力，不再作为科室业务主入口。
4. 分组、班次、规则、排班执行、本科室应用权限均可在排班应用完成。
5. 相关文档、测试清单、联调流程全部更新到新模型。
