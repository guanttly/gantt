## 一、问题陈述

### 1.1 用户反馈的核心痛点

> "员工没有机构相关信息，我在鼓楼医院建立了一个员工，那这个员工的层级是院级的，然后我想调整员工组织架构，分配到放射科，但是我却既看不到员工当前机构，也没办法调整员工到放射科。"
>
> "排班也有分组逻辑，它和行政架构有关系么？科室能跨很多组排班，下沉到组只能排本组的班？科室排完了本组还能不能排？权限混乱。"
>
> "一人多组是有可能的，比如A既承担CT/MR审核常规班，也需要承担急诊班。"

翻译为具体问题：

| 编号 | 问题 | 影响 |
|------|------|------|
| P1 | 员工列表没有直观展示所属组织的层级路径 | 管理员无法一眼看出员工属于哪个科室 |
| P2 | 员工详情/编辑页面缺乏组织调动功能入口 | 虽然后端支持修改 `org_node_id`，但前端未突出此能力 |
| P3 | 院级创建的员工，默认挂在医院根节点 | 后续需要手动调动到具体科室，但流程不清晰 |
| P4 | 科室管理员新增员工时，层级归属规则不明确 | 员工到底归属科室还是重新回到院级？管理员困惑 |
| P5 | 缺乏"人员调动"的专用操作和审计记录 | 调动等同于普通编辑，没有业务语义 |
| P6 | 行政架构层级未固化定义 | node_type 含义模糊，没有明确的层级约束 |
| P7 | 排班分组与行政架构的关系混乱 | 分组到底是行政单元还是排班单元？职责不清 |
| P8 | 排班权限没有按层级收口 | 非叶子节点管理员能排班导致权限溢出 |
| P9 | 一人多组场景下跨组冲突无检测 | 同一员工在多个排班组，可能被排到时间重叠的班次 |

### 1.2 现有系统的实际能力分析

| 能力 | 后端状态 | 前端状态 | 问题 |
|------|---------|---------|------|
| 员工绑定 org_node_id | ✅ 字段存在 | ⚠️ 表格仅显示节点名，无层级路径 | 看不出"放射科"属于哪个院区 |
| 编辑时修改 org_node_id | ✅ PUT 支持 | ⚠️ 编辑弹窗中有下拉框但不够醒目 | 用户不知道这就是"调动" |
| 调动后清理应用角色 | ✅ 后端自动清理 | ❌ 无提示 | 用户不知调动会重置权限 |
| 组织树查询 | ✅ GET /admin/org-nodes | ✅ 前端加载树 | 功能正常 |
| 排班分组 | ✅ employee_groups 表存在 | ✅ 分组管理页面 | 分组仅用于分配角色，不参与排班 |
| 排班以 org_node 为单位 | ✅ schedules.org_node_id | ✅ | 无法按分组排班 |
| 排班不区分分组 | ❌ 缺少 group_id | ❌ | 候选人取全节点，无法按组排 |

## 二、核心设计：行政架构与排班分组双轨模型

### 2.1 设计原则

**行政架构和排班分组是两个不同维度，必须分离。**

| 维度 | 行政架构（org_nodes） | 排班分组（employee_groups） |
|------|---------------------|--------------------------|
| 回答的问题 | 这个人归谁管？ | 这个人参与哪些排班轮转？ |
| 数据结构 | 树形，固定层级 | 扁平，挂在叶子节点下 |
| 员工关系 | **单归属**（1人→1节点） | **多归属**（1人→N个分组） |
| 决定什么 | 管理权限边界、数据可见范围 | 排班候选人范围 |
| 典型例子 | 张三是放射科的人 | 张三在CT审核组排白班，在急诊值班组排夜班 |

**为什么不能合并？** 因为一人可以参与多个排班轮转（如"A既承担CT/MR审核常规班，也需要承担急诊班"），但行政归属必须唯一。组织树强制单父节点，无法表达多归属。

### 2.2 行政架构：固定层级定义

行政架构的 `node_type` 必须遵循固定层级顺序，不允许跳级或乱序：

```
层级 0: organization（机构）  — 最顶层，如 "鼓楼医院"
层级 1: campus（院区）        — 可选，如 "本部院区"、"江北院区"
层级 2: department（科室）    — 排班管理的核心单元，如 "放射科"、"心内科"
```

**层级约束规则**：

| 规则 | 说明 |
|------|------|
| organization 下只能建 campus 或 department | 没有院区概念的医院可以直接建科室 |
| campus 下只能建 department | 院区下不能再建院区 |
| department 是行政架构的叶子节点 | 科室下不再建下级行政节点 |
| 员工的 org_node_id 必须指向叶子节点（department） | 不允许员工挂在机构或院区层级 |

> 注："员工必须在叶子节点"是**目标状态**。新建员工时如果尚未分配科室，暂挂在 organization 节点，标记为"待分配"，需管理员调动到具体科室后才能参与排班。

**层级示意**：

```
鼓楼医院 (organization) ←── 机构管理员管这里
├── 本部院区 (campus)    ←── 院区管理员管这里（如果有）
│   ├── 放射科 (department)  ←── 科室管理员 + 排班在这里
│   ├── 心内科 (department)  ←── 科室管理员 + 排班在这里
│   └── 急诊科 (department)  ←── 科室管理员 + 排班在这里
└── 江北院区 (campus)
    └── 内科 (department)
```

### 2.3 排班分组：挂在科室（叶子节点）之下

排班分组由科室管理员创建，属于科室内部的排班组织方式。分组**不是行政层级**，不出现在 org_nodes 树中。

```
放射科 (department)                    ← 行政架构的叶子节点
  ├── [排班分组] CT/MR审核组            ← 张三、李四、王五
  ├── [排班分组] 急诊值班组             ← 张三、赵六、钱七（注意：张三同时在两个组）
  ├── [排班分组] DR常规组              ← 李四、孙八
  └── （未分组员工：刚调入还没加组的）
```

**分组核心规则**：

| 规则编号 | 规则 | 说明 |
|---------|------|------|
| G1 | 分组归属于一个科室（叶子节点） | `employee_groups.org_node_id` 指向 department |
| G2 | 一个员工可以加入同一科室下的多个分组 | CT审核 + 急诊值班 |
| G3 | 排班以分组为单位 | 排班计划必须指定 group_id |
| G4 | 未加入任何分组的员工不参与排班 | 需先分组后排班 |
| G5 | 跨组时间冲突必须检测 | 张三在CT审核组排了白班，急诊组就不能给他同时段排班 |
| G6 | 只有科室管理员可以创建和管理分组 | 上级（机构/院区）管理员不直接管理分组 |

### 2.4 员工行政归属规则

**原则：员工的 `org_node_id` 必须指向科室（department），表示其行政隶属关系。**

| 规则编号 | 规则描述 | 说明 |
|---------|---------|------|
| R1 | 员工创建时必须指定所属科室 | 默认为创建者当前所在的叶子节点 |
| R2 | 机构管理员创建员工时可暂挂院级 | 标记为"待分配"，需后续调动到科室 |
| R3 | 科室管理员创建员工 → 自动归属该科室 | 只能在自己管辖的科室创建 |
| R4 | 只有机构根管理员可以调整员工的行政归属 | 跨科调动必须由上级操作 |
| R5 | 科室管理员不能调出本科室员工 | 只能管理科室内部的分组分配 |
| R6 | 调动到新科室后，原科室的排班分组成员关系自动清除 | 需在新科室重新加入分组 |
| R7 | 调动到新科室后，原科室的应用角色自动清除 | 需在新科室重新分配权限 |
| R8 | 员工的工号在所属科室内唯一 | 调动后工号在目标科室不得重复 |

### 2.6 排班流程：以分组为单位

```
第一步：机构管理员建行政架构
  └── 鼓楼医院 → 本部院区 → 放射科

第二步：录入员工 → 分配到科室
  └── 张三 → org_node_id = 放射科

第三步：科室管理员建排班分组
  ├── CT/MR审核组
  └── 急诊值班组

第四步：科室管理员分配员工到分组
  ├── 张三 → 加入 CT/MR审核组 + 急诊值班组（一人多组）
  ├── 李四 → 加入 CT/MR审核组
  └── 赵六 → 加入 急诊值班组

第五步：排班（以分组为单位）
  ├── CT/MR审核组 → 创建 3月排班 → 候选人：张三、李四
  └── 急诊值班组  → 创建 3月排班 → 候选人：张三、赵六
      └── ⚠️ 排班引擎检测到张三已在CT审核组有班次
          → 自动避开张三已排班的时段
          → 或提示冲突由排班员手动处理
```

### 2.7 跨组冲突检测

**核心规则：同一员工在同一时段只能有一个班次，无论来自哪个分组。**

```
场景 1: 时段不冲突 ✅
  CT审核组：张三 3/28 白班 08:00-17:00
  急诊组  ：张三 3/28 夜班 17:00-08:00+1  → ✅ 允许

场景 2: 时段完全重叠 ❌
  CT审核组：张三 3/29 白班 08:00-17:00
  急诊组  ：张三 3/29 白班 08:00-17:00    → ❌ 冲突，阻止

场景 3: 时段部分重叠 ❌
  CT审核组：张三 3/30 白班 08:00-17:00
  急诊组  ：张三 3/30 中班 12:00-20:00   → ❌ 冲突，阻止

场景 4: 不同组无共同成员 ✅
  CT审核组：李四 3/31 白班
  急诊组  ：赵六 3/31 白班               → ✅ 不同人，无冲突
```

**检测时机**：排班引擎 Generate 阶段，对每个候选人的每个待分配时段，查询该员工在**所有排班计划**中的已有班次，判断时段是否重叠。

### 2.8 人员生命周期

```
创建员工 ──→ [归属科室] ──→ [加入排班分组] ──→ [参与排班]
                │                 │                │
                │              移出分组          调动到新科室
                │                 │                │
                │                 ▼                ▼
                │         [不再参与该组排班]   [自动退出所有分组]
                │                              [自动清除应用角色]
                │                              [在新科室重新加入分组]
                │
             停用/离职
                │
                ▼
          [inactive，清除所有应用角色，退出所有分组]
```

## 三、具体改动

### 3.1 数据库变更

#### 3.1.1 schedules 表增加 group_id

```sql
ALTER TABLE schedules
    ADD COLUMN group_id VARCHAR(64) DEFAULT NULL COMMENT '排班分组ID，NULL 表示对科室下未分组人员排班',
    ADD INDEX idx_schedules_group (group_id);
```

#### 3.1.2 employee_app_roles 表增加 scope_group_id

```sql
-- 分组级排班员的权限范围约束
ALTER TABLE employee_app_roles
    ADD COLUMN scope_group_id VARCHAR(64) DEFAULT NULL
        COMMENT '限制权限生效范围到指定分组，NULL 表示科室全范围';
```

#### 3.1.3 org_nodes 增加层级约束校验

不改表结构，在后端 Service 层增加创建节点时的层级校验：

```go
// 允许的父子层级关系
var allowedChildTypes = map[string][]string{
    "organization": {"campus", "department"},  // 机构下可建院区或科室
    "campus":       {"department"},            // 院区下只能建科室
    "department":   {},                        // 科室是叶子，不能建下级
}

func (s *Service) validateNodeHierarchy(parentType, childType string) error {
    allowed := allowedChildTypes[parentType]
    for _, t := range allowed {
        if t == childType { return nil }
    }
    return fmt.Errorf("不允许在 %s 下创建 %s 类型节点", parentType, childType)
}
```

#### 3.1.4 建议增加的外键约束

```sql
ALTER TABLE employees
    ADD CONSTRAINT fk_employees_org_node
    FOREIGN KEY (org_node_id) REFERENCES org_nodes(id)
    ON DELETE RESTRICT ON UPDATE CASCADE;
```

#### 3.2.4 跨组冲突检测

排班引擎在为每个候选人分配班次前，检查该员工在**所有排班计划**中的现有时段：

```go
func (engine *Engine) checkCrossGroupConflict(ctx context.Context, empID, date string, newShift *Shift) error {
    existing, _ := engine.assignmentRepo.FindByEmployeeAndDate(ctx, empID, date)
    for _, a := range existing {
        existShift, _ := engine.shiftRepo.GetByID(ctx, a.ShiftID)
        if timeOverlap(existShift, newShift) {
            return fmt.Errorf("员工在 %s 已有排班（%s），与当前班次时段冲突",
                date, existShift.Name)
        }
    }
    return nil
}
```

#### 3.2.5 排班权限校验增强

```go
func (h *Handler) Create(w http.ResponseWriter, r *http.Request) {
    // 1. 检查 schedule:create 权限
    // 2. 检查当前用户所在节点是否为叶子节点（department）
    node := tenant.GetOrgNode(ctx)
    if node.NodeType != "department" {
        respondError(w, 403, "只有科室级管理员可以创建排班")
        return
    }
    // 3. 检查 scope_group_id 约束
    appRole := getAppRole(ctx)
    if appRole.ScopeGroupID != nil && *appRole.ScopeGroupID != input.GroupID {
        respondError(w, 403, "您只能为所负责的分组创建排班")
        return
    }
}
```

### 3.4 FAQ

#### Q1: 机构管理员在院级创建员工，该员工算哪个层级？

**A**: 员工暂挂在机构根节点，标记为"待分配"。这是过渡状态，适合批量导入后再逐批调动到各科室的场景。**待分配员工不能参与排班**（因为没有加入任何分组）。

#### Q2: 科室管理员创建的员工会不会层级混乱？

**A**: 不会。科室管理员通过 `resolveTargetOrgNodeID()` 被限制只能在本科室创建员工。创建后员工自动归属该科室，科室管理员再把员工加入排班分组即可。

#### Q3: 一个员工能属于多个分组吗？

**A**: 可以。行政归属唯一（一个科室），但排班分组多归属（如"张三既在CT审核组排白班，也在急诊值班组排夜班"）。排班引擎通过跨组冲突检测保证同一时段不会重复排班。

#### Q4: 科室排完了，分组还能排吗？

**A**: 不存在"科室排班"这个概念了。所有排班都以分组为单位。如果科室只有一个分组（比如小科室不需要细分），那就一个分组涵盖全科室人员。

#### Q5: 员工调动后原来的排班数据怎么办？

**A**: 已发布的历史排班不受影响。调动会清除原科室的分组成员关系和应用角色 → 员工自然不再出现在原科室任何分组的排班候选名单中。

#### Q6: 如果一个人没加入任何分组会怎样？

**A**: 不参与排班。前端分组管理页面会提示"未分组员工：N人"，引导管理员处理。

#### Q7: 非叶子节点管理员（机构/院区）为什么不能排班？

**A**: 排班是具体科室内部的业务操作，涉及班次细节、人员分组、时间冲突等，必须由了解本科室情况的人来执行。机构/院区管理员的职责是行政管理（建架构、调人、分配权限），不是具体排班。这避免了"上级排了班、科室又排了班"的冲突问题。

| 编号 | 验收项 | 通过条件 |
|------|--------|---------|
| AC-01 | 行政层级固化 | organization 下只能建 campus 或 department；campus 下只能建 department；department 下不能建子节点 |
| AC-02 | 员工列表组织路径展示 | "所属组织"列显示完整层级路径，如"鼓楼医院 / 本部院区 / 放射科" |
| AC-03 | 待分配标签 | 归属 organization 节点的员工在列表中显示"待分配"标签 |
| AC-04 | 调动功能 | 机构管理员可通过调动对话框将员工从一个科室调到另一个科室 |
| AC-05 | 调动副作用 | 调动后自动清除原科室的分组成员关系和应用角色，并在前端提示 |

| AC-08 | 排班以分组为单位 | 创建排班计划时必须选择一个排班分组，候选人来自该分组成员 |

| AC-10 | 分组排班员权限收口 | 拥有 `scope_group_id` 的排班员只能为指定分组创建排班 |
| AC-11 | 一人多组 | 同一员工可加入同一科室下的多个排班分组 |
| AC-12 | 跨组冲突检测 | 同一员工在不同分组被排到时间重叠的班次时，系统检测并阻止冲突 |
| AC-13 | 未分组员工不参与排班 | 未加入任何分组的员工不出现在排班候选人列表中 |
| AC-14 | 未分组员工提示 | 分组管理页面显示未分组员工数量和名单 |
| AC-15 | 工号冲突校验 | 调动员工到目标科室时，若工号重复则阻止 |
| AC-16 | 调动审计 | 调动操作写入审计日志，含原节点、目标节点、操作人 |
