# M06 排班管道 — 旧代码清理清单

> 适用范围：旧 FSM、历史 `schedule_v1` / `schedule_v2` / `schedule_v3`、当前 `schedule_v4` 以及相关兼容适配层。

## 一、原则

旧排班逻辑最终目标是删除，但**不能在 M06 落地后立即删除**。
必须按“冻结旧逻辑 → 对比验证 → 灰度切换 → 再清理”的顺序推进。

## 二、立即执行项

1. **冻结旧逻辑**
   - 旧目录只允许修复阻塞性缺陷。
   - 禁止继续向旧入口新增业务能力。

2. **建立对比测试**
   - 使用同一组 fixture，同时验证旧 `schedule_v4` 的核心语义与新 `Pipeline + Step` 的输出一致。
   - 优先覆盖：固定排班、排他规则、人员来源、兜底填充、全量校验。

3. **补运行指标**
   - 生成耗时
   - 排班总数
   - 违规总数
   - 规则命中数
   - 新旧 diff 数

4. **接入 shadow 模式**
   - 正式结果仍可由旧逻辑返回。
   - 新 Pipeline 后台并行执行，仅记录差异，不直接生效。

## 三、允许删除旧代码的门槛

只有同时满足以下条件，才允许删除旧代码：

- 兼容性测试全部通过；
- `deterministic` 与旧 V4 在核心场景下结果一致；
- 自动排班、手动调整、发布、变更记录四条链路均已通过集成验证；
- shadow 运行观察期内无高频差异；
- 数据迁移脚本、回滚方案、旧表备份策略已经就绪。

## 四、建议清理顺序

1. 停止旧入口继续接入新功能；
2. 删除旧调度入口与路由绑定；
3. 删除旧执行器、FSM、兼容适配层；
4. 删除废弃文档与遗留脚本；
5. 旧表先重命名备份，观察期结束后再 Drop。

## 五、当前已落地项

- 新排班骨架已存在于 `internal/core/schedule/step` 与 `internal/core/schedule/pipeline`；
- 已补基础兼容性测试，覆盖：
  - 固定排班规则；
  - 排他规则候选人过滤；
  - 人员来源规则候选人过滤。

## 六、当前运行时引用结论

代码扫描后，当前运行态已经收敛到 `schedule` 与 `schedule_v4/create`，V2/V3 旧链路已从运行时入口下线：

- `agents/rostering/internal/adapter/mapper.go`
   - 当前排班创建路径已固定到 `schedule_v4.create`；
   - 不再根据用户偏好分流到 `schedule_v2.create`、`schedule_v3.create`。
- `agents/rostering/internal/wiring/container.go`
   - 当前只注册 `schedule` 与 `schedule_v4/create`；
   - 仍注册 `ruleEngine` 供 `schedule_v4` 使用。
- `agents/rostering/internal/workflow/schedule_v4/create/actions_scheduling.go`
   - 仍通过 `ruleEngine` 服务实例驱动 `V4Executor` 执行正式排班；
   - 说明 `schedule_v4/executor` 不是历史残留，而是当前运行时路径的一部分。

结论：

- `v2 / v3` 已不再是运行时在线目录；
- 可以继续按“引用扫描 + 构建验证”的方式物理删除不可达旧目录。

## 十一、仓库级清理目标扩展

当前清理目标已经不再局限于 `agents/rostering/internal/workflow`，而是扩展到仓库级旧逻辑收缩：

- `cmd`
- `mcp-servers`
- `agents`
- `sdk`
- `services`

这几个目录不能按“目录名”直接删除，必须按“是否仍被当前部署主路径使用”来区分。

### A. 当前主路径，不能删

- `agents/rostering`
- `sdk/context`
- `sdk/rostering`
- `mcp-servers/context`
- `mcp-servers/rostering`
- `services/management-service`
- `cmd/agents/rostering`
- `cmd/mcp-servers/context-server`
- `cmd/mcp-servers/rostering-server`
- `cmd/services/management-service`

原因：

- 根目录 [Dockerfile](Dockerfile) 当前仍直接构建这些 `cmd/*` 入口；
- `agents/rostering/internal/wiring/container.go` 当前仍直接依赖 `sdk/context` 与 `sdk/rostering`；
- 上述目录组成当前多服务部署主路径，不能作为“旧代码”直接移除。

### B. 已确认可进入删除候选的仓库级孤立项

- `config/mcp-servers/data-server.yml`
- `config/mcp-servers/relational-graph-server.yml`

原因：

- 当前仓库中已无对应实现目录；
- 无 Docker 构建、无运行时入口、无代码直接引用；
- 仅属于历史配置残留。

### C. 当前应处理的仓库级历史入口

- `cmd/server`

原因：

- 它原本是历史单体入口；
- 当前已完成脚本脱钩并删除入口文件，后续只需保留清理记录。

### D. 不应误判为旧目录的当前主实现

- 根目录 `internal`

原因：

- 新架构的 `core / ai / auth / tenant / infra / platform` 当前主实现就在该目录下；
- 它不是历史单体残留，不能按“旧代码目录”清理；
- 后续只应清理其中真实的兼容分支或废弃子模块。

### E. 下一批仓库级清理动作

1. 再清理 V2/V3 存量状态常量与历史兼容引用；
2. 更新仍描述旧规则协议或旧版本切换的仓库文档；
3. 继续按引用扫描结果分批删除真实不可达代码。
