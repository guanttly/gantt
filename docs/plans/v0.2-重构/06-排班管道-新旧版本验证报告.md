## 三、仍未完成的关键验证

目前**还不能据此直接删除旧代码**，因为仍缺少“同输入、同输出”的双引擎对比验证：

- 使用同一组 fixture 同时驱动旧 `V4Executor` 与新 `deterministic pipeline`；
- 比较最终 assignments 是否一致；
- 比较 violations / conflicts 是否一致；
- 比较调整链路、发布链路、变更记录链路是否一致。

人员来源规则的一个已知缺口已经补齐：

- 新版 `SourceChecker` 已注册到 [internal/core/rule/checker/checker.go](internal/core/rule/checker/checker.go)；
- 共享 fixture 已新增 `source_validation` 场景，并由新版 `FullValidationStep` 消费；
- 目前已能确认：来源规则不仅参与候选过滤，也会进入新版全量校验结果。

但旧版来源规则走依赖解析链，仍需补一个更完整的双引擎同输入对比场景，不能据此直接认定新旧在来源依赖上完全等价。

进一步检查后确认：旧版 V4 当前的依赖解析器实际上只消费 `required_together`。

- [agents/rostering/internal/workflow/schedule_v4/executor/dependency_resolver.go](agents/rostering/internal/workflow/schedule_v4/executor/dependency_resolver.go) 中的 `findRequiredTogetherDeps()` 会显式过滤，只接受 `RuleTypeRequiredTogether`；
- 虽然旧版 `dependency_analyzer` 能分析出 `DependencyTypeSource`，但当前 resolver 不会执行这类依赖；
- 这意味着“旧版来源依赖”本身就不是完整落地状态，不能把它当作新版来源校验的权威对照实现。

因此，旧代码清理前的判断应调整为：

- 旧版可作为 `fixed / underfill / 部分排他语义` 的参考基线；
- `source` 相关能力要以新版设计目标和共享 fixture 为主验证，而不是要求与旧版行为完全一致。

另外，本轮验证中已发现并修复新版一个真实缺陷：

- [internal/core/schedule/step/phase_two.go](internal/core/schedule/step/phase_two.go) 原先只避免“同班次重复”，未避免“同一天跨班次重复分配”；
- 在排他场景下，这会让 `PhaseTwo` 重新把同一个人填入第二个班次；
- 现已修复为按“当天已占位”过滤候选人，并已通过共享 `exclusive_semantics` fixture 在新旧两侧验证。
