# AI 智能排班 SaaS 平台 — 重构总体方案

> **文档版本**: 1.0  
> **文档日期**: 2026-03-24  
> **关联文档**: [PRD](prd.md)  
> **状态**: 规划中

---

## 一、重构目标

将现有的「面向单一机构的智能排班框架」重构为「面向多租户的 SaaS 排班平台」。

**核心交付物**：

1. 单体模块化后端（Go），合并 7 个微服务为 3 个内部模块
2. 全新前端（Vue 3），消除所有历史兼容代码
3. 原生多租户（组织树 + GORM Scope），全链路数据隔离
4. Pipeline + Step 排班管道，替代 FSM 状态机
5. AI 可选增强，确定性排班引擎零 LLM 依赖

## 三、阶段计划

### 3.1 阶段一：地基搭建（第 1~4 周）

**目标**：搭建可运行的项目骨架，所有后续模块在此基础上开发。

| 周次 | 交付内容 | 涉及模块 |
|------|---------|---------|
| W1 | Go 项目骨架 + 数据库连接 + Redis + 路由框架 + 日志 | M01 |
| W1-W2 | org_nodes 表 + 组织树 CRUD + GORM 全局 Scope + 防漏 Callback | M02 |
| W2-W3 | JWT 认证 + RBAC + 组织节点切换 + 路由中间件 | M03 |
| W3-W4 | 员工/班次/分组/请假 CRUD API + 前端登录页 + 管理页骨架 | M04 + M10 |

**阶段一验收标准**：
- [ ] 当前多服务入口可分别启动成功：`cmd/services/management-service`、`cmd/mcp-servers/context-server`、`cmd/mcp-servers/rostering-server`、`cmd/agents/rostering`
- [ ] 注册 → 登录 → 获取 JWT → 带 Token 调用 CRUD API 全流程通过
- [ ] 不同组织节点登录后只能看到本节点数据
- [ ] GORM 防漏 Callback 拦截无 `org_node_id` 的查询

### 3.2 阶段二：排班核心（第 5~8 周）

**目标**：完成确定性排班引擎，可以在无 AI 的情况下完成完整排班。

| 周次 | 交付内容 | 涉及模块 |
|------|---------|---------|
| W5-W6 | 规则引擎（规则 CRUD + 规则继承合并 + 约束检查） | M05 |
| W5-W6 | Pipeline + Step 框架 + 确定性排班 Pipeline + 排班调整 Pipeline | M06 |
| W7 | 甘特图 + 表格视图 + 排班创建向导 | M07 |
| W8 | 请假管理完善 + 规则管理前端（继承标记） + 集成测试 | M05 + M07 |

**阶段二验收标准**：
- [ ] 创建排班计划 → 确定性引擎自动排班 → 甘特图展示 全流程通过
- [ ] 100 规则 × 50 人 × 7 天 排班 < 500ms
- [ ] 规则继承与覆盖：机构规则自动对下级科室生效，科室可覆盖
- [ ] 排班调整后实时校验规则冲突

### 3.3 阶段三：AI 增强（第 9~11 周）

**目标**：叠加 AI 能力，不影响确定性排班的独立运行。

| 周次 | 交付内容 | 涉及模块 |
|------|---------|---------|
| W9 | AI HTTP 适配器 + 配额管控 + 模型配置 API | M08 |
| W10 | AI 对话交互 + 意图识别 + AI 辅助规则解析 | M08 |
| W11 | AISelect Step + AI 辅助排班 Pipeline 组装 + 集成测试 | M08 + M06 |

**阶段三验收标准**：
- [ ] AI 功能开关关闭时，排班引擎正常工作
- [ ] AI 对话可查询/调整排班（自然语言）
- [ ] AI 辅助排班 Pipeline 产出方案质量 ≥ 确定性 Pipeline
- [ ] AI 调用日志和配额计数正确

### 3.4 阶段四：平台能力（第 12~14 周）

**目标**：完成平台管理功能，具备 SaaS 运营能力。

| 周次 | 交付内容 | 涉及模块 |
|------|---------|---------|
| W12 | 组织树管理后台 + 账户管理 | M09 |
| W13 | 订阅计费 + 审计日志 | M09 |
| W14 | 数据导入导出 + 统计看板 | M09 + M10 |

**阶段四验收标准**：
- [ ] 平台管理员可创建/停用机构，配置 AI 模型
- [ ] 审计日志记录所有敏感操作
- [ ] 统计看板展示工作量分布、规则合规率

### 3.5 阶段五：上线准备（第 15~16 周）

**目标**：生产就绪。

| 周次 | 交付内容 | 涉及模块 |
|------|---------|---------|
| W15 | 性能测试 + 安全审计 + 数据迁移脚本 | M11 |
| W16 | 文档 + 灰度发布 + 旧数据迁移验证 | M11 |

**阶段五验收标准**：
- [ ] 并发 500 租户压测通过
- [ ] 旧系统数据迁移脚本通过验证（数据完整性 100%）
- [ ] 安全扫描无高危漏洞

## 四、服务合并映射

### 4.1 现有服务 → 新模块映射

| 现有服务/组件 | 处理方式 | 目标模块 |
|-------------|---------|---------|
| Management Service（CRUD） | 直接迁移 | Core Service → `internal/core/` |
| Rostering Agent（规则引擎/排班算法） | 提取 V4 确定性逻辑 | Core Service → `internal/core/schedule/engine/` + `step/` |
| Rostering Agent（AI 意图识别/对话） | 剥离到 AI Service | AI Service → `internal/ai/` |
| Data Server | 删除，Core Service 直连 DB | 无（消除） |
| Context Server | 删除，对话管理内建 | AI Service → `internal/ai/chat/` |
| Relational Graph Server | 删除，规则引擎代码化 | Core Service → `internal/core/rule/` |
| Scheduling Service | 删除，与 Agent 合并 | Core Service |
| pkg/workflow/ FSM 引擎 | 删除，Pipeline 替代 | Core Service → `internal/core/schedule/pipeline/` |
| V1~V3 工作流定义 | 删除 | 无（消除） |
| V4 工作流定义 | 提取确定性逻辑到 Step | Core Service → `internal/core/schedule/step/` |
| 外部 auth-server | 删除，内建认证 | Admin Service → `internal/auth/` |

### 4.2 代码迁移路径

```
旧代码                                    新代码
───────────────────────────────────────────────────────────────
agents/rostering/internal/
  workflow/schedule_v4/
    v4_executor.go          ──→  internal/core/schedule/step/phase_*.go
    rule_engine.go          ──→  internal/core/schedule/engine/
  domain/model/             ──→  internal/core/ 各子模块 model.go
  internal/wiring/          ──→  历史单体入口 `cmd/server/main.go`（已删除）

services/management-service/
  内部 handler/service/repo  ──→  internal/core/ 各子模块

mcp-servers/rostering/
  tool/                     ──→  删除（Core Service 直连 DB）

pkg/ai/                     ──→  internal/ai/adapter/
pkg/config/                 ──→  internal/infra/config/
pkg/ws/                     ──→  internal/infra/websocket/

frontend/web/src/           ──→  web/src/（清理后迁移）
```

## 五、技术决策记录

| 编号 | 决策 | 决定 | 理由 | 替代方案 |
|------|------|------|------|---------|
| ADR-01 | 排班编排模式 | Pipeline + 可复用 Step | 排班是线性流程；Step 天然可复用；新增流程只需组合已有 Step | FSM 状态机（过重）、BPMN 引擎（过重） |
| ADR-02 | 内部通信 | 进程内函数调用 | 单体模块化，无需序列化开销 | gRPC（团队规模不支撑）、MCP（已验证不适合） |
| ADR-03 | 租户隔离 | GORM 全局 Scope + Callback 防漏 | 应用层统一拦截，零遗漏风险 | 数据库级隔离（成本高）、手动 WHERE（易遗漏） |
| ADR-04 | AI 调用 | 统一 HTTP 适配器直连大模型 API | 简单直接，无中间层 | MCP 协议（已验证过度抽象） |
| ADR-05 | 前端路由守卫 | 内建 JWT + 真实路由拦截 | SaaS 产品必须有完整认证 | 外部 auth-server（依赖外部） |
| ADR-06 | 数据库选型 | 仅 MySQL + Redis | 团队熟悉，减少运维负担 | + Neo4j（冗余）、+ Milvus（不再需要） |

## 六、风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|---------|
| V4 规则引擎逻辑提取不完整 | 排班结果不正确 | 中 | 编写对比测试：旧引擎和新 Step 输入相同数据，输出必须一致 |
| 数据迁移丢失 | 客户数据损失 | 低 | 迁移脚本分三步：备份 → 迁移 → 校验；保留旧表 30 天 |
| Pipeline 框架扩展性不足 | 无法应对复杂分支场景 | 低 | Pipeline 支持条件 Step（`ConditionalStep`），按需跳过 |
| 多租户 Scope 性能影响 | 大表查询变慢 | 中 | `org_node_id` 建联合索引；避免子查询，改为预查询节点 ID 列表 |
| AI 服务不可用 | 排班功能降级 | 低 | 确定性 Pipeline 完全独立于 AI，降级透明 |
| 前端重构工作量超预期 | 延期 | 中 | 前端与后端并行开发；按页面优先级排序，非核心页面延后 |

## 七、质量保障

### 7.1 测试策略

| 层级 | 覆盖目标 | 工具 |
|------|---------|------|
| 单元测试 | 每个 Step、每个规则处理器 | Go testing + testify |
| 集成测试 | Pipeline 端到端、API 端到端 | Go testing + httptest |
| 对比测试 | 新旧引擎输出一致性 | 自定义对比框架 |
| 性能测试 | 排班引擎 < 500ms、API < 200ms | k6 / wrk |
| E2E 测试 | 前端核心流程 | Playwright |

### 7.2 代码质量

- CI 流水线：lint（golangci-lint / eslint）→ 单元测试 → 构建 → 集成测试
- 每个模块完成后进行代码评审
- 关键模块（租户隔离、排班引擎、认证）需两人评审
